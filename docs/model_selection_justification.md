# Обґрунтування вибору моделей

У роботі порівнюються три регресійні підходи для прогнозування `latency` та `energy` нейронних архітектур на edge-пристроях:

- `Linear Regression`;
- `HistGradientBoostingRegressor`;
- `MLPRegressor`.

Такий набір моделей обрано не випадково. Він покриває три різні рівні складності: просту лінійну baseline-модель, нелінійну ансамблеву модель для табличних даних і невелику нейронну мережу. Це дозволяє не лише знайти найкращу модель за метриками, а й показати, наскільки задача потребує нелінійного моделювання.

## Чому використано Linear Regression

`Linear Regression` використано як базову модель для порівняння. Вона перевіряє просту гіпотезу: чи можна описати latency лінійною комбінацією ознак, таких як пристрій, dataset і кількість операцій в архітектурі.

Для `NASBench201 -> latency` модель отримала:

| Model | MAE | RMSE | MAPE, % |
| --- | ---: | ---: | ---: |
| Linear Regression | 3.9790 | 6.0433 | 455.17 |
| Gradient Boosting | 0.8746 | 2.1589 | 187.25 |
| MLP Neural Network | 0.8700 | 2.1571 | 185.89 |

Linear Regression працює приблизно у `4.55` раза гірше за MAE, ніж найкраща модель:

```text
3.9790 / 0.8700 ≈ 4.55
```

Причина полягає в тому, що latency не є лінійною функцією від архітектурних ознак. Вплив операцій залежить від пристрою: одна й та сама операція може мати різну вартість на `raspi4`, `fpga`, `eyeriss` або `edgetpu`. Крім того, між ознаками є взаємодії: наприклад, кількість `nor_conv_3x3` впливає на latency не ізольовано, а разом із типом пристрою, dataset і загальною структурою архітектури. Лінійна модель не може достатньо добре описати такі взаємодії без явного додавання складних перехресних ознак.

Отже, Linear Regression у роботі потрібна як baseline, який показує, що проста лінійна залежність для цієї задачі недостатня.

## Чому використано Gradient Boosting

`HistGradientBoostingRegressor` обрано як сильну модель для табличних даних. Вона добре підходить для задач, де:

- є числові та категоріальні ознаки після кодування;
- залежності між ознаками нелінійні;
- важливі взаємодії між ознаками;
- потрібна стабільна якість без дуже складного налаштування.

Для цієї задачі Gradient Boosting суттєво покращив результат порівняно з Linear Regression:

```text
MAE: 3.9790 -> 0.8746
```

Це означає, що модель змогла краще врахувати нелінійні залежності між архітектурою, dataset і пристроєм. Також Gradient Boosting є практично зручним baseline для табличних ML-задач, бо зазвичай добре працює навіть без великої кількості даних для кожної окремої комбінації ознак.

## Чому використано MLP Neural Network

`MLPRegressor` використано як нейромережеву модель, що може апроксимувати складні нелінійні залежності. Її включено до порівняння, оскільки тема роботи пов'язана з нейронними мережами, а сама задача прогнозування latency потенційно має складну нелінійну структуру.

Для `NASBench201 -> latency` MLP показала найменший MAE:

```text
Gradient Boosting MAE: 0.8746
MLP MAE: 0.8700
Різниця: 0.0047
```

Різниця між Gradient Boosting і MLP мінімальна:

```text
(0.8746 - 0.8700) / 0.8746 ≈ 0.54%
```

Тому не можна стверджувати, що MLP принципово краща. Коректний висновок такий: для цього експерименту MLP має найкращий числовий результат, але Gradient Boosting практично не поступається їй і залишається стабільною альтернативою.

## Чому різниця між Gradient Boosting і MLP мала

Різниця між Gradient Boosting і MLP невелика, тому що обидві моделі здатні описувати нелінійні залежності. У цьому наборі даних ключові закономірності значною мірою визначаються табличними ознаками:

- `device`;
- `dataset`;
- кількість різних операцій в архітектурі;
- кількість `nor_conv_3x3`, `skip_connect`, `none` тощо.

Gradient Boosting добре працює з такими табличними ознаками. MLP також здатна моделювати складні залежності, але її перевага стає суттєвою лише тоді, коли дані мають складніші приховані представлення або коли вхідні ознаки містять більше інформації про структуру архітектури. У поточній реалізації архітектура описана агрегованими ознаками, тому Gradient Boosting уже витягує більшу частину доступного сигналу.

## Остаточне обґрунтування вибору

У роботі доцільно залишити всі три моделі:

- Linear Regression потрібна як проста baseline-модель і показує, що задача не зводиться до лінійної залежності.
- Gradient Boosting є основною сильною моделлю для табличних даних і дає значне покращення якості.
- MLP Neural Network перевіряє, чи дає нейромережева апроксимація додатковий виграш.

За результатами `NASBench201 -> latency` найкращою за MAE є `MLPRegressor`, але її перевага над `HistGradientBoostingRegressor` мінімальна. Тому в практичному висновку слід зазначати не лише "найкращу" модель, а й те, що Gradient Boosting має майже таку саму якість і може розглядатися як стабільна та інтерпретованіша альтернатива для табличних benchmark-даних.

## Чому на сайті використовуються дві моделі

У Streamlit-інтерфейсі для рекомендацій одночасно використовуються дві збережені моделі:

- модель для прогнозування `latency`;
- модель для прогнозування `energy`.

Це зроблено тому, що `latency` і `energy` є різними цільовими змінними. Вони описують різні характеристики виконання нейронної мережі: час inference та енергоспоживання. Для них можуть відрізнятися розподіли значень, набір доступних рядків у даних і найкраща модель за метрикою MAE.

Наприклад, у поточних експериментах для `latency` найкращою стала `MLPRegressor`, а для `energy` - `HistGradientBoostingRegressor`. Тому сайт не використовує одну універсальну модель для всіх показників. Він завантажує окрему найкращу модель для кожної цілі:

```text
*_latency_predictor.pkl -> predicted_latency
*_energy_predictor.pkl  -> predicted_energy
```

Такий підхід є коректним для задачі рекомендацій, тому що конфігурація має одночасно задовольняти обмеження за двома різними критеріями. Спочатку окрема модель прогнозує latency, друга модель прогнозує energy, а потім модуль рекомендацій фільтрує та сортує конфігурації за цими двома прогнозованими значеннями.

## Чому для latency і energy обрані різні моделі

Результати експериментів показали, що одна й та сама модель не є найкращою для всіх цільових змінних.

| Search space | Target | Best model | MAE |
| --- | --- | --- | ---: |
| NASBench201 | latency | MLPRegressor | 0.8700 |
| NASBench201 | energy | HistGradientBoostingRegressor | 1.0956 |
| FBNet | latency | MLPRegressor | 0.5622 |
| FBNet | energy | HistGradientBoostingRegressor | 2.5714 |

Це сталося з кількох причин.

По-перше, `latency` і `energy` описують різні фізичні характеристики. Latency відображає час виконання inference, а energy - енергоспоживання. Вони пов'язані між собою, але не є однаковими залежностями від архітектури та пристрою.

По-друге, для `energy` доступно менше навчальних даних. У HW-NAS-Bench значення energy є не для всіх пристроїв, тому energy-моделі навчаються на меншій підмножині записів. Наприклад, у NASBench201 для latency використано `267153` рядки, а для energy - `140625` рядків. Менший обсяг даних може змінювати те, яка модель працює стабільніше.

По-третє, energy має інший розподіл значень і сильніше залежить від конкретної hardware-платформи. Для таких табличних залежностей Gradient Boosting часто працює стабільніше, бо добре розбиває простір ознак на локальні правила. Для latency MLP отримала трохи кращий результат, оскільки змогла краще апроксимувати плавні нелінійні залежності між архітектурними ознаками та часом виконання.

Отже, різні найкращі моделі для latency і energy є очікуваним результатом. У проекті модель обирається не вручну, а за мінімальним MAE окремо для кожної цільової змінної.
